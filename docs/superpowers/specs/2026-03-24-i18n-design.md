# i18n Design — English + German

**Date:** 2026-03-24
**Status:** Approved
**Scope:** Full app (landing pages + authenticated app)

---

## Summary

Add English/German internationalisation to potmagic using `react-i18next` with TypeScript module augmentation for type-safe translation keys. Language is encoded in the URL (`/de/` prefix for German, no prefix for English) using TanStack Router's optional `($lang)` segment. A language switcher button appears in the landing navbar and the authenticated sidebar, next to the existing theme toggle.

---

## Decisions

| Question | Decision | Rationale |
| --- | --- | --- |
| Library | `react-i18next` | Battle-tested, SSR-friendly, type-safe via module augmentation |
| Language storage | URL prefix | SEO-friendly, shareable links, no hydration mismatch |
| English URL | No prefix (`/stories`) | English is default; `/de/stories` for German |
| Type-safety | TypeScript module augmentation | Zero extra tooling; keys typed against `en.json` |
| Namespace strategy | Single flat namespace with feature prefixes | Simple; two languages don't warrant multi-namespace |
| `show/$storyId` | Excluded from lang subtree | Broadcast URLs are shared externally — adding a prefix breaks existing links |

---

## 1. Routing Architecture

### Optional `($lang)` segment

All routes **except `show/`** move into a `($lang)/` folder under `src/routes/`. TanStack Router's optional param syntax means a single route tree handles both `/concept` and `/de/concept`. The `show/` route family stays as a sibling of `($lang).tsx` at the root level, keeping broadcast URLs stable.

**Target file structure:**

```text
src/routes/
  __root.tsx                          (unchanged)
  ($lang).tsx                         ← layout: validates lang, sets i18next language
  ($lang)/
    index.tsx                         →  /  and  /de/
    concept.tsx                       →  /concept  and  /de/concept
    pricing.tsx
    docs.tsx
    docs/
      index.tsx
      create-story.tsx
      add-scenes.tsx
      props.tsx
      size-guidelines.tsx
    auth.tsx
    auth.director-setup.tsx
    under-construction.tsx
    _app.tsx                          ← route ID changes to /($lang)/_app (see note below)
    _app/
      profile.tsx
      director.tsx
      stories/
        index.tsx
        $storyId/
          index.tsx
          scenes/$sceneId.tsx
      stage/
        index.tsx
        $sceneId.tsx
  show/                               ← stays at root — NOT under ($lang)
    index.tsx
    $storyId.tsx
```

**Important:** Moving `_app.tsx` under `($lang)/` changes its TanStack Router route ID from `/_app` to `/($lang)/_app`. The `createFileRoute` call inside `_app.tsx` must be updated to match:

```ts
// Before
export const Route = createFileRoute('/_app')({ ... })

// After
export const Route = createFileRoute('/($lang)/_app')({ ... })
```

The same applies to every child route inside `_app/` — their route IDs gain the `($lang)` prefix. Verify each `createFileRoute` call after the move.

### `($lang).tsx` layout route

This layout validates the `lang` param, changes the i18next language on the client, and passes language through router context for use in `head()` meta tags.

```ts
// src/routes/($lang).tsx
import { createFileRoute, redirect, Outlet } from '@tanstack/react-router'
import { useEffect } from 'react'
import i18n from '@/i18n'

export const Route = createFileRoute('/($lang)')({
  beforeLoad: ({ params }) => {
    const lang = params.lang  // undefined = English, 'de' = German
    // Only reject explicitly unknown lang values — undefined (EN) is valid
    if (lang !== undefined && lang !== 'de') {
      throw redirect({ to: '/' })
    }
    return { locale: lang ?? 'en' }
  },
  component: function LangLayout() {
    const { lang } = Route.useParams()
    const locale = lang ?? 'en'

    useEffect(() => {
      // changeLanguage is client-only — avoids SSR singleton mutation (see SSR note)
      i18n.changeLanguage(locale)
      document.documentElement.lang = locale
    }, [locale])

    return <Outlet />
  },
})
```

**SSR note — i18next singleton:** `i18next` is a module-level singleton. Calling `i18n.changeLanguage()` inside `beforeLoad` on the server is unsafe under concurrent requests — the second request's language call would corrupt the first. The solution here is to call `changeLanguage` only in a client-side `useEffect`. On the server, i18next stays at its initialisation language (`'en'`). This means server-rendered HTML always uses English strings; the client hydrates and immediately updates to the correct language. For this project's scale this trade-off is acceptable. If full SSR localisation is needed later, migrate to a per-request i18next instance using `createInstance()`.

### Validation logic — why `undefined` must pass

The condition `lang !== undefined && lang !== 'de'` is intentional. With an optional param:

- URL `/stories` → TanStack Router resolves `lang = undefined`, child path `stories` → correct match
- URL `/de/stories` → `lang = 'de'`, child path `stories` → correct match
- URL `/fr/stories` → `lang = 'fr'` → redirect to `/`

The previous (incorrect) condition `lang !== 'en' && lang !== 'de'` would have redirected English users (`lang = undefined`) to `/`. Never check for `'en'` — English is represented as `undefined` (no prefix in URL).

```sh

```

### File structure

```text
src/

    en.json         ← source of truth; TypeScript typed against this
    de.json         ← German translations (same keys)
    index.ts        ← i18next initialisation
```

### Type-safety via module augmentation

```ts
// src/i18n/i18next.d.ts
import en from './en.json'

declare module 'i18next' {
  interface CustomTypeOptions {
    defaultNS: 'translation'
    resources: {
      translation: typeof en
    }
  }
}
```

After this, `t('nav.pricig')` is a TypeScript compile error. Editor auto-complete lists all valid keys.

### Initialisation

```ts
// src/i18n/index.ts
import i18n from 'i18next'
import { initReactI18next } from 'react-i18next'
import en from './en.json'
import de from './de.json'

i18n.use(initReactI18next).init({
  lng: 'en',           // default; overridden client-side by ($lang).tsx useEffect
  fallbackLng: 'en',
  resources: {
    en: { translation: en },
    de: { translation: de },
  },
  interpolation: { escapeValue: false },  // React escapes by default
})

export default i18n
```

Import this file once at the app entry point (the TanStack Start client bootstrap file).

## 3. `useLanguage` Hook

A single hook for all components. Avoids direct `react-i18next` imports spreading through the codebase.

```ts
// src/hooks/useLanguage.ts
import { useTranslation } from 'react-i18next'
import { useParams } from '@tanstack/react-router'

export function useLanguage() {
  const { t } = useTranslation()
  const { lang } = useParams({ strict: false })
  const locale = lang ?? 'en'
  const langPrefix = locale === 'de' ? '/de' : ''

  return { t, locale, langPrefix }
}
```

**On `langPrefix` and type safety:** `langPrefix` is a raw string helper for convenience. Using it in TanStack Router `<Link to={...}>` bypasses route type-checking. Prefer using `useNavigate()` with full typed paths where possible, and reserve `langPrefix` for cases where typed `to` props are not feasible (e.g. dynamic external links, meta tag URLs). Typos in paths using `langPrefix` will be runtime 404s, not TypeScript errors.

Usage:

```tsx
const { t, langPrefix } = useLanguage()

// Preferred — typed route
<Link to="/$lang/stories" params={{ lang }}>
  {t('nav.stories')}
</Link>

// Acceptable — untyped but convenient
<Link to={`${langPrefix}/stories`}>{t('nav.stories')}</Link>
```

## 4. Translated `head()` Meta Tags

Route `head()` functions run server-side where `useTranslation()` is unavailable. Pass the resolved locale through route context from `beforeLoad` and read it in `head()`:

```ts
// src/routes/($lang)/index.tsx
export const Route = createFileRoute('/($lang)/')({
  head: ({ match }) => {
    const locale = match.context?.locale ?? 'en'
    const isDE = locale === 'de'
    return {
      meta: [
        { title: isDE ? 'potmagic: Live Theater' : 'potmagic: Live Story Theater' },
        { name: 'description', content: isDE ? 'German description...' : 'English description...' },
        // hreflang alternates for SEO
        { tagName: 'link', rel: 'alternate', hreflang: 'en', href: 'https://potmagic.com/' },
        { tagName: 'link', rel: 'alternate', hreflang: 'de', href: 'https://potmagic.com/de/' },
      ],
    }
  },
  component: LandingPage,
})
```

Each public route should include `hreflang` alternate links pointing to both the English and German versions. This is required for correct Google indexing of the two-language site. For routes with dynamic segments (e.g. `/stories/$storyId`), construct the full URL from match params.

**`__root.tsx` head conflict:** The root route's `head()` has a hardcoded English `title` and `description`. These act as fallbacks but will bleed through on German routes unless overridden. Remove `title` and `description` from `__root.tsx`'s `head()` and move them into each public route's `head()` with locale-aware values. Keep only `charSet`, `viewport`, `og:site_name`, `og:type`, and `twitter:card` at the root level since those are language-neutral.

## 5. Language Switcher Component

```tsx
// src/components/language-switcher.component.tsx
import { useNavigate, useRouterState } from '@tanstack/react-router'
import { useLanguage } from '@/hooks/useLanguage'

export function LanguageSwitcher() {
  const { locale } = useLanguage()
  const navigate = useNavigate()
  const pathname = useRouterState({ select: (s) => s.location.pathname })

  const toggle = () => {
    if (locale === 'en') {
      // /concept → /de/concept
      navigate({ to: `/de${pathname}` })
    } else {
      // /de/concept → /concept  |  /de → /
      const stripped = pathname.replace(/^\/de/, '') || '/'
      navigate({ to: stripped })
    }
  }

  return (
    <button
      type="button"
      onClick={toggle}
      className="btn btn-ghost btn-sm btn-square font-display text-xs tracking-wider"
      aria-label="Switch language"
    >
      {locale === 'en' ? 'DE' : 'EN'}
    </button>
  )
}
```

**Button label convention:** shows the *target* language (what you switch *to*), matching the sun/moon theme toggle convention.

### Placement

- **`landing-navbar.component.tsx`** — immediately left of the theme toggle, in both desktop and mobile menus
- **`sidebar.component.tsx`** — next to the theme toggle at the bottom of the sidebar

### Edge case behaviour

| Current URL | Switch to | Result |
| --- | --- | --- |
| `/` | DE | `/de/` |
| `/de` (no slash) | EN | `/` |
| `/concept` | DE | `/de/concept` |
| `/stories/abc123` | DE | `/de/stories/abc123` |
| `/de/stage/xyz` | EN | `/stage/xyz` |
| `/de/stories/abc123` | EN | `/stories/abc123` |

## 6. `<html lang>` Attribute

`RootDocument` is the `shellComponent` of `__root.tsx` and wraps the entire match tree. It cannot use `useRouteContext` (which requires being inside a matched route) or `useRouterContext()` (which does not exist in `@tanstack/react-router` v1). The correct approach is `useRouterState` with a matches selector:

```tsx
// src/routes/__root.tsx — RootDocument
function RootDocument({ children }: { children: ReactNode }) {
  const locale = useRouterState({
    select: (state) => {
      const langMatch = state.matches.find((m) => m.routeId === '/($lang)')
      return (langMatch?.params as { lang?: string })?.lang ?? 'en'
    },
  })

  return (
    <html lang={locale} suppressHydrationWarning>
      <head><HeadContent /></head>
      <body>{children}<Scripts /></body>
    </html>
  )
}
```

This reads the lang param from the matched route list — safe in both SSR and client contexts. On the server, `lang` will be `undefined` (English) for the initial render of `/de/` routes because `changeLanguage` is deferred to client `useEffect`, so `<html lang>` will render as `"en"` server-side and update to `"de"` on hydration. `suppressHydrationWarning` (already present for theme) suppresses the React mismatch warning for this attribute.

## 7. Migration Steps

> **Important:** Do this work on a dedicated branch. Step 4 (moving routes) breaks the app until steps 5–8 are complete. Don't merge incrementally.

1. Install packages: `pnpm add react-i18next i18next`
2. Create `src/i18n/` with `en.json`, `de.json`, `index.ts`, `i18next.d.ts`
3. Import `src/i18n/index.ts` in the TanStack Start client entry point
4. Move all route files (except `show/`) into `src/routes/($lang)/`
5. Create `src/routes/($lang).tsx` layout with validation and `useEffect` locale sync
6. Update every `createFileRoute` path string in moved files to include `/($lang)` prefix — especially `_app.tsx` (`/_app` → `/($lang)/_app`) and all its children
7. Create `src/hooks/useLanguage.ts`
8. Create `src/components/language-switcher.component.tsx`
9. Update `landing-navbar.component.tsx` — add `<LanguageSwitcher />`, update all `<Link>` `to` props, replace hardcoded strings with `t()`
10. Update `sidebar.component.tsx` — add `<LanguageSwitcher />`, update links and strings
11. Translate all hardcoded strings in all routes and components into `en.json` / `de.json`
12. Update `head()` meta tags in each public route to use locale-aware strings and `hreflang` alternate links
13. Update `__root.tsx` `RootDocument` to read locale from router context for `<html lang>`
14. Verify: `/stories` loads correctly (lang = undefined), `/de/stories` loads correctly (lang = 'de'), `/fr/stories` redirects to `/`
